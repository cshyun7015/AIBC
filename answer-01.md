제시해주신 과제 가이드라인을 바탕으로, 실제 엔터프라이즈 환경에서 바로 활용할 수 있는 수준의 실무형 **'지능형 ITSM 인시던트 해결 Multi-Agent'** 기획안을 작성했습니다.

과제의 요구사항(Multi-Agent, RAG, 서비스 패키징)을 모두 충족하면서도, MSA 환경 및 모니터링 시스템과의 연동을 고려한 고도화된 시나리오입니다.

# **[AntiGravity ITSM 지능형 인시던트 분석 Agent – 이름 입력]**

## **1. 프로젝트 개요 – 기획 배경 및 핵심 내용**

### **1.1 프로젝트 기획 배경**

* 현재 독립적인 마이크로서비스(MSA) 아키텍처로 구성된 시스템 환경에서는 장애(Incident) 발생 시 여러 서비스 간의 호출 추적 및 근본 원인 파악이 매우 복잡합니다.
* 기존 방식은 운영자가 일일이 모니터링 대시보드를 확인하고 과거 티켓 이력을 수동으로 검색해야 하므로 평균 복구 시간(MTTR)이 길어지는 한계가 있습니다.
* 복잡한 컨테이너 환경의 로그와 과거 해결 지식을 AI가 직접 분석하여 즉각적인 트러블슈팅 가이드를 제공하는 Agent 서비스로 이 Pain Point를 해결하고자 합니다.

### **1.2 핵심 아이디어 및 가치 제안(Value Proposition)**

* **핵심 기능:** 사용자가 장애 증상을 입력하면, Multi-Agent가 과거 티켓(RAG)과 실시간 시스템 상태를 종합 분석하여 원인과 해결 코드를 제시합니다.
* **제공 가치:** 단순 Q&A가 아닌, 실제 시스템 인프라 및 애플리케이션 에러에 대한 '실행 가능한(Actionable) 해결책'과 '검증용 E2E 테스트 시나리오'를 자동 생성하여 업무 효율을 극대화합니다.
* **차별성:** 단일 LLM에 의존하지 않고, 이슈 분류(Triage) ➔ 원인 분석(Root Cause) ➔ 품질 검증(QA)으로 이어지는 에이전트 간(A2A) 협업 파이프라인을 구축합니다.

### **1.3 대상 사용자 및 기대 사용자 경험(UX)**

* **주요 타겟:** L1/L2 IT 운영 담당자, 데브옵스(DevOps) 엔지니어, 백엔드/프론트엔드 개발자
* **기대 경험:** 포털 화면에 장애 현상("요청 관리 모듈 응답 지연")을 텍스트로 입력하는 것만으로, 원인 분석 리포트와 수정 후 검증할 수 있는 테스트 스크립트까지 One-Stop으로 제공받습니다.

## **2. 기술 구성 – 서비스에 적용할 기술 스택**

### **2.1 Prompt Engineering 전략**

* **역할 기반 프롬프트:** 'L1 헬프데스크', 'L2 시스템 아키텍트', 'QA 엔지니어'로 각 에이전트의 페르소나를 명확히 분리하여 전문적인 응답을 유도합니다.
* **CoT (Chain of Thought):** 장애 원인을 추론할 때 '증상 확인 ➔ 관련 로그 검색 ➔ 과거 사례 대조 ➔ 결론 도출'의 사고 과정을 명시하도록 프롬프트를 설계합니다.
* **출력 구조화:** 모든 에이전트의 최종 응답은 JSON 형태(원인, 해결책, 위험도, 테스트 코드)로 강제하여 시스템 간 데이터 교환의 안정성을 확보합니다.

### **2.2 LangChain / LangGraph 기반 Agent 구조**

* **Multi-Agent 설계:** LangGraph를 활용하여 상태(State)를 공유하며 순환 피드백을 주고받는 3개의 핵심 에이전트로 구성합니다.
* **Triage Agent (분류):** 사용자의 요청을 분석하여 인프라 문제인지, 애플리케이션 문제인지 1차 분류합니다.
* **Root-Cause Agent (분석):** RAG 검색과 도구(Tool)를 활용하여 실제 장애 원인을 분석하고 해결책을 제시합니다.
* **QA-Master Agent (검증):** 해결책이 적용된 후 정상 작동을 확인하기 위한 Playwright 기반의 라이프 사이클 E2E 테스트 스크립트를 생성합니다.

### **2.3 RAG 구성**

* **데이터 파이프라인:** 과거 마리아DB에 적재된 ITSM 인시던트 티켓 내용과 해결 이력, 그리고 MSA 아키텍처 가이드 문서를 수집하여 텍스트로 변환합니다.
* **임베딩 및 DB:** 텍스트 분할(Chunking) 후 임베딩 모델을 통해 벡터화하고, FAISS 또는 ChromaDB에 저장하여 빠르고 정확한 의미 기반 검색을 구현합니다.

### **2.4 서비스 개발 및 패키징 계획**

* **프론트엔드 (UI):** React와 TypeScript를 활용하여 대화형 채팅 UI 및 결과물(코드, 리포트)을 렌더링하는 깔끔한 대시보드를 구성합니다.
* **백엔드 (API):** Spring Boot를 메인 애플리케이션 서버로 두고, LangGraph가 구동되는 AI 코어 로직은 FastAPI로 분리하여 마이크로서비스 형태로 연동합니다.
* **배포 환경:** Docker 컨테이너 환경으로 패키징하여 이식성과 확장성을 확보합니다.

### **2.5 선택적 확장 기능**

* **A2A (Agent-to-Agent) 협업:** Root-Cause 에이전트가 내놓은 해결책을 QA 에이전트가 검토하고, 테스트가 불가능한 구조일 경우 다시 Root-Cause 에이전트에게 수정을 요청하는 순환 구조(Cyclic Edge)를 설계합니다.
* **MCP (Model Context Protocol) 연동:** 에이전트가 외부 모니터링 스택(Grafana/Loki/Promtail)의 로그 조회 API나 컨테이너 상태 조회 도구를 직접 호출할 수 있도록 Tool Calling을 고도화합니다.

## **3. 주요 기능 및 동작 시나리오**

### **3.1 사용자 시나리오(Use Case Scenario)**

* **Step 1:** 사용자가 포털에서 "요청 관리 서비스에서 '요청 등록' 후 '요청 조회' 시 500 에러가 간헐적으로 발생함"이라고 입력합니다.
* **Step 2:** Triage Agent가 문제를 분석하고 애플리케이션 로직 에러로 분류하여 Root-Cause Agent에게 전달합니다.
* **Step 3:** Root-Cause Agent가 RAG를 통해 과거 유사한 '조회 API 타임아웃/트랜잭션 락' 사례를 검색하고 해결 가이드(DB Connection 튜닝 방안)를 도출합니다.
* **Step 4:** QA-Master Agent가 해당 문제를 검증하기 위해 "요청 등록 ➔ DB 확인 ➔ 요청 조회 ➔ 요청 삭제" 라이프 사이클을 확인하는 Playwright 컨테이너 테스트 코드를 자동 작성합니다.
* **Step 5:** UI에 최종 해결 리포트와 검증용 테스트 코드가 구조화되어 출력됩니다.

## **5. 추가 아이디어 (선택)**

* **자동 롤백 및 적용 (Auto-Remediation):** 분석 단계에 머물지 않고, 에이전트가 안전한 범위 내에서 컨테이너 설정을 직접 수정하고 테스트를 돌려보는 자동 치유(Auto-healing) 파이프라인으로 확장할 수 있습니다.
